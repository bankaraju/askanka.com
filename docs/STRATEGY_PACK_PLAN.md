## The architectural pattern: two-repo strangler

| Repo | Purpose | Continues to evolve? |
|---|---|---|
| `askanka.com` (this repo) | Research playground; full 172 deps; experimental hypotheses; daily working environment | Yes — but with commercial-attitude discipline (see below) |
| `anka-strategy-pack` (NEW) | Production-grade extraction; ~8 deps; reproducible backtests; shippable as tarball | Yes — but only via promotion gate |

**Both repos process the same upstream data.** Both compute Phase C / H-001 / NEUTRAL filter / ETF regime independently. **A daily diff job at 16:30 IST EOD** compares their outputs:

- **Identical:** extraction is canonical, the pack is shippable
- **Diverge by even one row:** halt pack release, root-cause the divergence, fix, re-diff

This is the strangler pattern from microservices extraction — proven, conservative, and the only way to be sure the pack isn't a slow drift away from the validated research.

## Commercial-attitude discipline (in `askanka.com` from now on)

Every NEW file or feature gets one of two tags from day one:

### `pipeline/research/` — research-only

- One-off scripts, hypothesis exploration, data prep, ad-hoc analysis
- Fine to be messy, fine to have personal paths, fine to be coupled to your laptop
- Never ships in the pack
- No requirement for tests, docs, or external-reader voice

### `pipeline/strategy_pack/` — strategy-pack-candidate

- Held to the higher bar from the moment it lands
- Required: tests, frozen data, external-reader docstring, no personal paths, no Bharat-isms in comments
- Sibling-imports forbidden (must use `from pipeline.foo import bar`)
- Must be reachable via the dependency archaeology script (i.e., `pipeline/strategy_pack/` is added to the seed list)

### Promotion gate (research → strategy_pack)

A research file can be promoted to strategy_pack only when it passes:
1. **Validated:** has a verdict.json showing pass on §9 / §9A / §9B / decade-split (when applicable)
2. **Frozen data:** input snapshot lives in the pack repo, hash-verified
3. **External-reader docs:** module-level docstring + at least one ELI5 explainer in `docs/strategy-explanations/<name>.md`
4. **Test coverage:** at least one happy-path test + one regression test
5. **Clean imports:** no sibling-imports, no implicit dependencies on personal config
6. **Two pairs of eyes:** Bharat reads, Claude (or future reviewer) signs off in PR

This is the **code governance policy** that's been missing from `GOVERNANCE_INDEX.md` Section 3. The promotion gate operationalizes it.

## Order routing (Tier 1 personal-live, post-pack)

The order routing piece is a follow-on, not a pre-requisite for the pack. The pack ships with a paper-mode-only order router; flipping to live happens after:

1. Pack is in a quant shop's hands and they've evaluated it
2. Order router has been hardened: max-position guards, daily loss limit, kill-switch file, emergency stop runbook
3. 30-day shadow run on Bharat's own Kite account, paper mode, against the existing paper engine — outputs diff'd, identical
4. Single-lot live test on Bharat's own account
5. Gradual scale-up

**For an IA partnership:** the pack ships to the IA partner with the paper-mode router. The IA's compliance team handles the live-routing layer on their side, integrating with their RA-approved client distribution. Bharat doesn't need to host the live execution stack.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Extraction drifts from research repo over time | Daily byte-diff job catches drift within 24h |
| Hidden dependency missed during cut | Dependency archaeology script re-run on every seed addition; CI fails if a non-pack module is reachable |
| Pack accumulates research bloat | Promotion gate (research → strategy_pack) is strict; PR review required |
| Quant shop evaluator can't reproduce numbers | Frozen data snapshots in pack; deterministic seeds; one-command backtest entry point |
| LLM dependencies block shipping | Gemma migration framework wired 2026-04-30; week-1 results inform whether to ship with Gemma or hold |
| Personal/proprietary data leaks into pack | `scripts/secrets_scrubber.py` (TBD) runs in CI on every PR; rejects commits with email addresses, telegram tokens, or hardcoded user paths |
