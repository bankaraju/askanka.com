# Sector-Correlation Stability Study — design 2026-04-30

**Status:** discovery-only research. No edge claim, no hypothesis-registry entry.
Anything we *act* on after this study registers a fresh single-touch hypothesis
per `docs/superpowers/specs/backtesting-specs.txt §10.4`.

## Question

Across the F&O sector universe (~23 sectors, ~276 tickers), are there
**stable** sector-pair correlations and lead-lag relationships in 5y daily
data that survive across multiple random sub-windows? And — beyond pairwise
— do *clusters* of sectors co-move in a way that constitutes an actionable
"basket signal" (up-side or down-side)?

The motivation is the live observation 2026-04-30 that Power & Utilities
dragged the H-001 cohort hard (-0.84%, 1/5W) while Telecom flew (+4.01%,
2/2W). We want to know whether that's a fluke or part of a structural
sector-pair relationship that has historical precedent — and whether
combinations of sectors moving together give earlier warning than any
single sector alone.

## Phase 0 — Data acceptance gate (CLAUDE.md mandatory)

**Source dataset:** `pipeline/data/fno_historical/<TICKER>.csv`. Daily OHLCV
~5y. Refreshed by `AnkaDailyDump` 04:30 IST per the canonical schedule.

The gate (per `docs/superpowers/specs/anka_data_validation_policy_global_standard.md`):

1. **Registration (§6):** dataset is registered as the F&O OHLCV panel
   feeding `canonical_fno_research_v3.json`. ✅
2. **Schema contract (§8):** columns must be `[Date, Close, High, Low,
   Open, Volume]`; Date must parse to ISO; no nulls in Close.
3. **Cleanliness gates (§9):**
   - Per-ticker row count must be ≥ 90% of the expected (≈1236 trading
     days for the 5y span 2021-04-23 → 2026-04-22). Tickers below this
     threshold are flagged and **excluded** from the study.
   - Stale-tail check: last bar must be within 5 trading days of
     today's last close. Stale tickers excluded.
4. **Adjustment mode (§10):** these prices are adjusted-close (yfinance
   default). Document this; we will use them as-is for return computation
   (log returns are split-invariant on adjusted prices).
5. **Point-in-time (§11):** the panel is *backward-looking only*. We are
   not training a forward predictor here, so PIT correctness is naturally
   satisfied (every observation is an after-the-fact fact).
6. **Sector-mapping completeness:** every ticker must map to a sector via
   `pipeline/scorecard_v2/sector_mapper.py`. Unmapped tickers are dropped
   from the sector-index construction.
7. **Small-sector handling:** sectors with fewer than 5 mapped tickers
   (currently `Business_Services` n=2, `Defence` n=5, `Real_Estate_Hotels`
   n=5, `Hospitals_Diagnostics` n=5) are flagged. We *include* them in
   the analysis but report results with a `small_sector=True` tag so the
   reader knows the index is fragile.

The audit report is written to
`pipeline/data/research/sector_correlation/data_audit_2026_04_30.md`
**before** any correlation is computed. If the gate fails (any §8 schema
violation), the study aborts and reports the failure.

## Phase 1 — Sector-index construction

For each sector, build an **equal-weighted total-return index** from the
constituent tickers' adjusted close:

1. Compute per-ticker daily log return `r_t = log(C_t / C_{t-1})`.
2. For each trading day `d`, sector return = simple mean of constituent
   returns available on day `d` (skip-NaN; document daily n).
3. Index level is the cumulative product `∏(1 + r_d)` rebased to 100 on
   the first day all sectors have ≥ 3 constituents.

Output panel: `sector_index_panel.parquet` — DataFrame indexed by date,
columns = sectors, values = daily log returns. This is the *single
artifact* every downstream analysis reads from.

## Phase 2 — Pairwise stability via bootstrap windows

For each pair (s_i, s_j) in the 23×22/2 = 253 unordered pairs:

1. **Full-period correlation:** Pearson on all ~1236 daily returns.
2. **Bootstrap stability:** draw 100 random 1-year windows (≈252 bars
   each) at random start offsets from the 5y span. Compute correlation
   in each. Report:
   - mean of |corr| across windows
   - fraction of windows with |corr| ≥ 0.5 (the **stability score**)
   - 95% bootstrap CI on the correlation
3. A pair is "stably correlated" if stability score ≥ 0.70.
4. A pair is "stably anti-correlated" if all of (mean corr ≤ -0.3) AND
   (fraction with corr ≤ -0.3 in window) ≥ 0.70.

Window choice rationale: 1y is long enough to span a typical regime
(roughly: COVID, 2022 bear, 2023 sideways, 2024 rally, 2025 mixed) and
short enough that 100 random offsets give us bootstrap variance. We
accept overlap between draws — overlap inflates apparent stability but
does *not* invalidate the rank ordering of pairs, which is what we use.

## Phase 3 — Lead-lag

For each pair, compute Pearson correlation at lags 0, +1, +2, +3 trading
days (sector i return at day d vs sector j return at day d+k). The
"leader" is the side that has a higher correlation at positive lag than
the contemporaneous correlation. Report only lead-lag relationships that
are themselves stable across the bootstrap windows (same 70% rule).

## Phase 4 — Up/down asymmetric correlation

Two pairs may have similar full-period correlation but co-move only on
up days or only on down days. Compute:

1. Conditional correlation on **up days** = days where the cross-sectional
   median sector return > 0.
2. Conditional correlation on **down days** = days where the median < 0.

Pairs with `|corr_up - corr_down| ≥ 0.3` are tagged `asymmetric`. These
are the ones that matter for the user's "early sign" framing — a pair
that's tightly correlated on down days but loose on up days is *only*
an early-warning signal in bear regimes.

## Phase 5 — Cluster / multi-sector basket discovery

Hierarchical-clustering on the 23×23 full-period correlation matrix
(distance = 1 - |corr|, average linkage). Cut the dendrogram at distance
0.5 and report the resulting clusters. For each cluster:

- Member sectors
- Mean within-cluster correlation
- Bootstrap stability of the cluster (fraction of windows where the same
  sectors cluster together at the same cutoff)

Clusters that are stable in ≥ 60% of windows are the candidate "basket
signals" — sectors that, when they all move together, signal something
the market is pricing as one factor.

## What this study does NOT do

- It does not produce a trading rule. Stable correlations ≠ alpha.
- It does not certify any pair for live execution. The §10.4 single-touch
  rule applies if/when we want to act on a finding.
- It does not address regime conditioning beyond up/down splits — a
  proper regime-conditional study would need the v3-curated regime tape
  joined onto the panel, which is out of scope for this discovery pass.

## Time budget

Coverage audit: 5 min. Index build: 10 min. Bootstrap correlation
(253 pairs × 100 windows): 30-45 min. Cluster + lead-lag + asymmetric:
15 min. Findings write-up: 10 min. Total target: 75-90 min.
