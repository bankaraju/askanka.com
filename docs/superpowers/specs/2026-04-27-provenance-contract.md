# Engine/Model Output Provenance Contract — 2026-04-27

**Status:** active. Producers opt-in. Consumers degrade gracefully when sidecar is absent.

**Why:** during the AnkaETFSignal v1 → v3_curated cutover on 2026-04-27, the docs said v3 was live but the scheduled task still pointed to v1 for several minutes. We had to monitor `schtasks /Query` to confirm the switch landed. A permanent in-UI badge ("regime label generated by: ETF v3 curated · 04:45 IST · 2m ago") would have made the ambiguity impossible. This contract makes the running system the source of truth, not the docs.

## Rule

Every model output (JSON, CSV, parquet, anything a downstream consumer reads) gets a sibling **provenance sidecar** written by the producing scheduled task at start time, *before* the output itself is written.

Sidecar path: `<output_path><.provenance.json>` — i.e., `data/today_regime.json` → `data/today_regime.json.provenance.json`.

Single canonical helper: `pipeline/provenance.py`. Producers import `write`, consumers import `read` or `assess`.

## Producer responsibility (opt-in, one-time per task)

At the very start of every scheduled-task entry point, before any heavy work:

```python
from pipeline import provenance

provenance.write(
    output_path="data/today_regime.json",
    task_name="AnkaETFSignal",
    engine_version="v3_curated",          # bump on cutover
    expected_cadence_seconds=86400,       # 24h for daily outputs
    extras={"input_etf_count": 31},       # optional task-specific
)
```

`git_sha` and `started_at` are auto-filled. Producers must not call `provenance.write` *after* the heavy work completes — the contract is "this task started running at this time with this version", not "this task succeeded".

## Consumer responsibility

Read the sidecar with `provenance.assess(output_path, expected_engine_version=...)` to get a UI-ready dict:

```python
{
    "engine_version": "v3_curated",
    "expected_engine_version": "v3_curated",
    "task_name": "AnkaETFSignal",
    "started_at": "2026-04-27T04:45:02+05:30",
    "output_age_seconds": 18412.0,
    "expected_cadence_seconds": 86400,
    "color": "green",         # green | amber | red | unknown
    "reason": "ok",
}
```

Render a badge from this. Color rules:
- **green** — version matches expected, output age within cadence
- **amber** — sidecar missing (producer not opted-in yet) OR output age > 1.5× cadence
- **red** — version mismatch with `expected_engine_versions.json` OR output age > 2× cadence

Click-to-expand reveals the full sidecar + inferred dependencies.

## Configuration: expected versions

`pipeline/config/expected_engine_versions.json` declares what each tracked output's `engine_version` should be today. The terminal/audit reads from here. **Update this file in the same commit as any engine cutover.** If the cutover landed but config wasn't updated, the badge goes red — that's the warning.

## Cutover protocol

When changing the engine version of an output (e.g. v1 → v3 curated):

1. Update `expected_engine_versions.json` with the new `expected_engine_version`.
2. Update the producing scheduled task's `provenance.write(engine_version=...)` call to match.
3. Update the .bat wrapper / `pipeline/scripts/anka_inventory.json` notes if the script path changed.
4. Commit all three changes together.
5. Wait for the next scheduled run; verify the badge goes green within one cadence.
6. If the badge is still red after one cadence, the cutover did not land — investigate before declaring victory.

The protocol explicitly puts the badge between "I think the cutover landed" and "the cutover has landed".

## What this is NOT

- Not a kill switch. The badge surfaces drift; it doesn't gate trades or stop tasks.
- Not a replacement for the watchdog (`pipeline/watchdog.py`). Watchdog flags missing/stale outputs by inventory; provenance flags wrong-version-but-fresh outputs and unknown-version outputs. Both are needed.
- Not retroactive. Outputs produced before this contract landed don't have sidecars. Don't backfill — the only honest answer is "we don't know which version generated yesterday's value."
