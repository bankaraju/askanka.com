# §5A Triage — H-2026-04-23-001 compliance run

**Run:** `pipeline/autoresearch/results/compliance_H-2026-04-23-001_20260423-150125/`
**Manifest data_audit classification:** `AUTO-FAIL` (impaired_pct = 10.349%)
**Triage verdict:** **DETECTOR NOISE — true impairment is ~0.21% (CLEAN).**
**Root cause:** `data_audit.audit_ticker` compares observed dates against a
generic `pd.bdate_range` (Mon–Fri) spanning the **global** min→max window of
all tickers. That produces two systematic false positives.

## 3. Corrected classification

Subtracting both false-positive classes:

```
residual_missing = long_hist_missing − (185 tickers × 69 NSE holidays)
                 = 12,773 − 12,765
                 = 8 bars
effective_total  = 258,816  (excludes pre-listing pseudo-bars)
true_impaired    = 8 + 50 stale + 494 zero_vol = 552
true_impaired_pct = 552 / 258,816 = 0.213%
→ classification: CLEAN
```

## 4. Implications

1. **H-2026-04-23-001 FAIL decision stands** on its own merits — the FAIL
   came from zero (ticker, direction) pairs surviving Bonferroni at 100k
   permutations, not from data impairment. The §5A `AUTO-FAIL` tag on the
   manifest is a false flag and does not affect the gate-checklist ruling.
2. **Every future compliance run will AUTO-FAIL on §5A** until the detector
   is fixed — this is a blocking issue for the TA pilot (H-2026-04-24-001)
   and every other strategy brought through the framework.
3. **Raw-bar canonicity gate is unaffected.** `execution_window.is_tradeable`
   only checks bars inside the trade's execution window (MODE A: T and T+1).
   NSE-holiday flags can still fire if T lands on a business day that happens
   to be a holiday — which is exactly the behavior we want (no trade on a
   non-trading day). The listing-effect false positives can also affect this
   gate for recently-listed tickers when the global `bdate_range` is passed
   in, so the fix below resolves both issues.

## 5. Recommended fix (tiny)

Two changes to `pipeline/autoresearch/overshoot_compliance/data_audit.py`
and one pass-through update in `runner.py`:

**a. Per-ticker listing window in `audit_ticker`.** Replace the external
`business_days` argument with an observed-based calendar derived from the
ticker's own first trading date onward, intersected with the global calendar:

```python
first_obs = pd.DatetimeIndex(df.index).min().normalize()
expected = set(dt for dt in pd.DatetimeIndex(business_days).normalize()
               if dt >= first_obs)
```

This alone collapses the 15,024 listing-effect phantoms to 0.

**b. Use the observed NSE calendar, not `bdate_range`.** Build the expected
calendar once per run from the union of dates observed by long-history
tickers (≥ 1000 bars), rather than `pd.bdate_range`. This encodes the exchange
holiday schedule without any external dependency:

```python
# in runner.py, Step 2 preamble
observed_dates = set()
for t, df in ticker_frames.items():
    if len(df) >= 1000:
        observed_dates.update(pd.DatetimeIndex(df.index).normalize())
bdays = pd.DatetimeIndex(sorted(observed_dates))
```

After these two changes, RELIANCE's 69 false "missing" and every
listing-effect phantom both drop to 0. Expected final `impaired_pct` ≈ 0.2%
(the 50 stale + 494 zero-volume bars), classification = `CLEAN`.

## 6. Scope

Fix is independent of the canonicity policy — it sharpens §5A detector
accuracy but does not change any gate thresholds, trade invalidation logic,
or metric computation. Safe to ship as a follow-up to `860e8ec`.
